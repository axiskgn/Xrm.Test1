Данная программа разрабатывалась как однопользовательская, но для реализации многопользовательского
сценария работы достаточно написать соответствующий набор репозиториев.

Про тесты: программа разрабатывалась на лету, это означает что архитектура менялась часто
по моему глубокому убеждению юнит тесты нуны для проверки алгоритмов и соответствия архитектуре
при разработки архитектуры они мешают и значительно повышают стоимость разработки.

При решении задачи для сокращения сроков разработки были приняты следующие решения:
1) резюме представляется только в первой рубрике из тех в которых оно находится;
2) механизм мержа новых данных упрощен до предела;
3) репозитарии не имеют кэша в памяти что увеличивает время загрузки данных с сайта 
(чем больше данных в БД тем медленнее) - решается данная проблема созданием кэширующей обёртки IRootRepository;
4) нет механизмов для сохранения заметок относительно людей и резюме
5) при повторной загрузке резюме с изменениями создаётся новая копия этого резюме (они объединяются через sourceInfo)
но в интерфейсе пользователя это никак не обрабатывается;
6) с портала загружается только 1000 резюме, но загрузку всех резюме реализовать не сложно,
хотя смысл это делать имеет только после оптимизации репозиториев (см. пункт 3)
7) нет возможности заменять текстовые значения enum-ов;
8) полное отсутствие оформления пользовательского интерфейса (в дальнейшем необходимо создать и применить стили).

Потенциал для развития и возможности для дальнейшей разработки после 
ликвидации допущений, принятых для сокращения времени разработки:
1) реализация поставщиков данных с других порталов;
2) реализация трехзвенной архитектуры;
3) улучшение интерфейса пользователя;
4) реализация гибкой системы запросов;
5) реализация возможности объединения резюме в группы по определенному признаку;
6) реализация возможности сравнения выбранных резюме;
7) реализация возможности просмотра истории редактирования резюме.
